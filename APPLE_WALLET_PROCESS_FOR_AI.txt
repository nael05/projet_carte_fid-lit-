Guide détaillé — Génération fiable de Pass Apple Wallet (pour IA / DevOps)

But: fournir une méthode reproductible, sécurisée et automatable pour générer des .pkpass Apple Wallet.
Langue: Français

---
1) Contexte et principes

- Un "pass" Apple Wallet est un fichier ZIP signé (.pkpass) contenant:
  - `pass.json` : métadonnées (formatVersion, passTypeIdentifier, teamIdentifier, etc.)
  - ressources : icônes (`icon.png`, `icon@2x.png`, `logo.png`...), autres images
  - `manifest.json` : table de hachages SHA1 des fichiers dans l'archive
  - `signature` (PKCS#7) : signature du manifest avec le certificat Pass Type ID

- Garantie de fiabilité = contrôle des étapes (création du manifest, signature correcte, packaging) + sécurisation des clés privées + tests automatisés.

---
2) Conditions préalables

- Un compte Apple Developer avec Pass Type ID créé (ex: pass.com.fidelyz.apple.passkit)
- Certificat Pass Type ID (exporté depuis le portail Apple en .p12)
- Serveur sécurisé (TLS) pour stocker le certificat et délivrer le .pkpass
- OpenSSL installé sur le serveur
- Outils Node.js disponibles (pour orchestration) ou scripts bash/powershell

---
3) Extraction et préparation des certificats

Objectif: partir du `.p12` Apple (PKCS#12), obtenir 2 PEM propres : `certificate.pem` (certificat) et `key.pem` (clé privée) sans métadonnées OpenSSL indésirables.

Commandes (sécuriser la passphrase si présente):

# Extraire la clé privée (PEM)
openssl pkcs12 -in apple-wallet-cert.p12 -nocerts -out key-encrypted.pem
# Si la clé est chiffrée et vous voulez la déchiffrer (attention aux permissions):
openssl rsa -in key-encrypted.pem -out key.pem
# Extraire le certificat (PEM)
openssl pkcs12 -in apple-wallet-cert.p12 -nokeys -out certificate.pem

Remarques:
- `key.pem` doit commencer par `-----BEGIN PRIVATE KEY-----` (ou `RSA PRIVATE KEY` selon le format).
- Supprimez ou protégez les fichiers intermédiaires. Attribuez `chmod 600` (Linux) ou permissions équivalentes.

---
4) Structure minimale de `pass.json` (exemple)

{
  "formatVersion": 1,
  "passTypeIdentifier": "pass.com.fidelyz.apple.passkit",
  "teamIdentifier": "8QYMJ4RJ55",
  "organizationName": "Fidelyz",
  "description": "Carte de fidélité Fidelyz",
  "serialNumber": "000001",
  "storeCard": {
    "primaryFields": [{"key":"points","label":"Points","value":"0"}],
    "secondaryFields": [{"key":"name","label":"Nom","value":"Member"}],
    "backFields": [{"key":"terms","label":"Conditions","value":"..."}]
  },
  "barcode": {
    "format":"PKBarcodeFormatQR",
    "message":"CLIENT-123456",
    "messageEncoding":"iso-8859-1"
  }
}

- Adapter le champ `serialNumber` pour chaque client (unique).
- `barcode` ou `barcodes` selon le modèle choisi.

---
5) Génération du manifest.json

- Le manifest est un objet JSON où chaque entrée est le sha1 hex digest d'un fichier contenu dans le pass.
- Exemple de script Node.js minimal (conceptual):

const fs = require('fs');
const crypto = require('crypto');
function sha1Hex(buffer) { return crypto.createHash('sha1').update(buffer).digest('hex'); }
const files = ['pass.json', 'icon.png', 'icon@2x.png', 'logo.png'];
const manifest = {};
for (const f of files) {
  manifest[f] = sha1Hex(fs.readFileSync(f));
}
fs.writeFileSync('manifest.json', JSON.stringify(manifest, null, 2));

---
6) Signer le manifest (PKCS#7) avec OpenSSL

- Apple demande un fichier de signature PKCS#7 (binary) signé avec le certificat Pass Type ID et la clé privée.
- Commande OpenSSL (exemples):

openssl smime -binary -sign -signer certificate.pem -inkey key.pem -certfile WWDR.pem -in manifest.json -out signature -outform DER

Remarques:
- `WWDR.pem` : Apple Worldwide Developer Relations CA certificate (télécharger depuis Apple et vérifier date).
- Le `-certfile` option inclut la chaîne d'intermédiaires; certains outils exigent que la chaîne WWDR soit incluse.
- Résultat : `signature` (binaire DER) — ce fichier est ajouté au ZIP sous le nom `signature`.

---
7) Packaging en .pkpass (ZIP)

- Les fichiers à inclure: `pass.json`, `manifest.json`, `signature`, et les images.
- L'ordre n'est pas strict mais évitez compression non déterministe; utilisez ZIP sans timestamps variables ou normalisez. Exemple avec `zip` CLI:

zip -j -X fidelyz-CLIENT-123.pkpass pass.json manifest.json signature icon.png icon@2x.png logo.png

Options:
- `-j` = junk paths, `-X` = eXclude extra file attributes (utile pour compatibilité)

---
8) Vérifications (avant envoi au client)

1. Vérifier que `pass.json` est valide JSON et respecte schema Apple.
2. Vérifier que `manifest.json` contient SHA1 correspondant (recalculer et comparer).
3. Vérifier que `signature` est valide (openssl can verify):

openssl smime -verify -in signature -inform DER -content manifest.json -noverify

Ce test vérifie que signature peut être décodée et le contenu récupéré.

---
9) Commande complète (script automatisé, exemple en bash)

# 1. préparer un dossier temporaire
mkdir /tmp/pass-tmp && cp pass.json icon.png icon@2x.png logo.png /tmp/pass-tmp/
cd /tmp/pass-tmp
# 2. générer manifest
node -e "const fs=require('fs');const crypto=require('crypto');const files=fs.readdirSync('.');let m={};files.forEach(f=>m[f]=crypto.createHash('sha1').update(fs.readFileSync(f)).digest('hex'));fs.writeFileSync('manifest.json',JSON.stringify(m));"
# 3. signer
openssl smime -binary -sign -signer /path/to/certificate.pem -inkey /path/to/key.pem -certfile /path/to/WWDR.pem -in manifest.json -out signature -outform DER
# 4. zipper
zip -j -X ../fidelyz-CLIENT-123.pkpass *
# 5. nettoyer
cd ..; rm -rf /tmp/pass-tmp

10) Intégration serveur (node.js express example)

- Endpoint POST /api/wallet/apple qui reçoit les données client et retourne le .pkpass
- Étapes du handler:
  1. Valider input
  2. Construire `pass.json` à partir du template (remplacer serialNumber, champs, barcode)
  3. Copier ressources dans un dossier temporaire
  4. Générer `manifest.json`
  5. Signer avec OpenSSL (exécuter la commande via child_process)
  6. ZIP en .pkpass
  7. Envoyer en `res.setHeader('Content-Type','application/vnd.apple.pkpass')` et `res.send(buffer)`

Sécurité:
- NE JAMAIS stocker la clé privée dans le code. Utiliser un répertoire protégé, permissions restreintes (600), ou un service KMS (AWS KMS, Azure KeyVault) pour signer.
- Limiter l'accès à l'API (auth, quotas)
- Auditer les accès et utiliser rotation de clés

11) Tests à automatiser

12) Déploiement & CI

- CI ajoute étapes:
  - Exécuter tests
  - Ne jamais stocker la clé privée dans CI runners (utiliser secret manager)
  - Option: déléguer signature à un job protégé/privé qui utilise OpenSSL et la clé privée (runner dédié)

---
13) Debugging fréquent

- Erreur "Cannot read properties of undefined (signerKeyPassphrase)": signe que la librairie JS utilisée n'accepte pas la structure du certificat; solution : utiliser OpenSSL pour signer.
- Erreur "Invalid PEM formatted message" : vérifier que PEM commence/termine correctement; supprimer lignes "Bag Attributes" et métadonnées OpenSSL.
- Problèmes de compatibilité : vérifier versions OpenSSL, vérifier encoding (binary vs text) lors de signature

---
14) Exemple de script Node.js complet (résumé)

- Génération de pass.json à partir de template
- Écrire fichiers dans temp dir
- Exécuter commande node pour manifest (ou calculer via crypto)
- Exécuter openssl via child_process.execSync
- zipper via archiver ou zip CLI
- renvoyer le buffer

---
15) Points opérationnels & sécurité

- Sauvegarde chiffrée des .p12 (use HSM/KMS when possible)
- Rotation: créer procédure pour révoquer et renouveler Pass Type ID
- Monitoring: journaux pour chaque signature et tentative

---
16) Annexe — commandes OpenSSL utiles

# Afficher contenu .p12
openssl pkcs12 -info -in apple-wallet-cert.p12
# Extraire cert
openssl pkcs12 -in apple-wallet-cert.p12 -nokeys -out certificate.pem
# Extraire clé (non chiffrée)
openssl pkcs12 -in apple-wallet-cert.p12 -nocerts -nodes -out key.pem
# Signer manifest
openssl smime -binary -sign -signer certificate.pem -inkey key.pem -certfile WWDR.pem -in manifest.json -out signature -outform DER

---
17) Conclusion

Cette méthode (manifest->openssl smime -> zip PKPASS) est la plus robuste et portable pour la production. Elle contourne les incohérences des bibliothèques Node.js et s'appuie sur OpenSSL (fiable et maintenu). Automatiser et sécuriser la clé privée est la clé du succès.
